TL;DR

Recommendation Summary

For a small-to-medium project that wants extensibility without overengineering, the sweet spot is a hybrid loading model (npm packages bundled at build time now, with a clean escape hatch for URL-loaded modules later), a React-first but Web-Component-compatible widget contract, and a typed widget SDK versioned from day one. Start small on the trust model and grow into sandboxing only if third-party contributors become real.

Widget Loading Model

How does the dashboard shell discover and load widget code? This is the most foundational choice — it affects build tooling, developer experience, runtime security, and how hard it is to add a new widget.

Option Pros Cons Complexity Best Fit Recommendation
npm packages
Widgets are installed dependencies
Full type-safety end-to-end Tree-shaking, standard tooling Offline, reproducible builds IDE autocomplete everywhere
Rebuild required per widget add External devs need npm publish flow Version-lock coupling to shell
Low Internal teams, CI-controlled deploys ✓ Start here for MVP
Remote URLs
Dynamic import() at runtime
No redeploy of shell needed Widget author controls rollout True runtime extensibility
Must share React singleton (import maps / externals) CSP and CORS complexity Network failures, version drift No type safety without extra tooling
Medium SaaS with many independent widget teams → Add in Phase 2
Plugin registry/manifest
Central catalog + lazy-load
Discoverable widget library Enable/disable without rebuild Metadata for config UI
Registry infrastructure needed Over-engineered for small projects Maintenance burden
High Marketplace / ecosystem products → Phase 3 only if ecosystem grows
Module Federation
Webpack 5 / Vite federation
Mature runtime-sharing solution Shared deps without bundling twice Works with existing npm + CI
Webpack lock-in (Vite plugin less stable) Complex config, hard to debug Version skew still possible
High Large orgs already on Module Federation → Skip unless forced
Hybrid
npm now, remote ESM escape hatch
Start simple, grow into runtime loading SDK contract defined upfront No forced re-architecture later
Requires disciplined widget interface from day 1 Remote path adds complexity when used
Low → Med Projects expecting growth, starting small ⭐ Recommended
💡 Key insight: Define a WidgetModule interface (default export with meta, component, configSchema) from day one. npm packages satisfy this interface at build time; remote URLs satisfy it at runtime. The shell doesn't care which delivery mechanism was used.
Decision 2

Trust Boundary

How much do you trust widget code? The answer determines whether widgets run in the main thread with full DOM access, or in a restricted sandbox.

Option Pros Cons Complexity Best Fit Recommendation
Trusted first-party
Widgets run in main thread, same origin
Zero overhead, full React integration No messaging boilerplate Direct context/state access Simple debugging
A bad widget can crash the whole app No isolation — full DOM access
Low Internal teams, known widget authors ⭐ Recommended for MVP
Sandboxed iframes
Each widget in a sandboxed iframe
Strong origin isolation Widget crash doesn't affect shell Works for 3rd-party code
Postmessage API for everything Theme sync is non-trivial Drag-and-drop across iframe boundaries is painful Accessibility challenges
High Marketplaces with unknown widget authors → Phase 3 if 3rd-party opens up
Capability-limited contract
Widgets only access host via typed API
Soft boundary without iframe cost Auditable surface area Easy to add CSP headers
Enforced by convention, not runtime Still same-thread execution
Medium Semi-trusted external developers → Good Phase 2 upgrade
Web Workers / Worklets
Widget logic off-thread
Logic isolated from UI thread Protects UI responsiveness
No DOM access — rendering must stay main thread Complex serialization Overkill for typical dashboard widgets
High CPU-intensive data widgets (charts, ML) → Opt-in per widget only
⚠️ Watch out: Adding iframe sandboxing after the fact is extremely disruptive — drag-and-drop, theming, React context, and host APIs all need rethinking. If third-party authors are even a medium-term possibility, define a capability API from day one so you're not passing raw React context to widgets.

Decision 4

Design System Baseline (MUI)

Should the project adopt Material UI (MUI) as a shared visual baseline, and if so, how deeply should it be coupled to widgets? The only hard requirement for a widget is that it is a React component conforming to the WidgetModule interface — nothing more. MUI access is a layered opt-in: the shell's <ThemeProvider> is already in the React tree, so a widget that installs @mui/material itself gets the full live theme automatically via useTheme(), while a widget that has no MUI dependency still gets visual consistency through CSS custom properties mirrored from the theme on :root.

Option Pros Cons Complexity Best Fit Recommendation
MUI everywhere
Shell + all widgets must use MUI
100 % visual consistency Shared theme object — no token mapping needed All MUI components available in every widget
Hard requirement for every external widget author MUI version coupling across all repos Emotion / styled-components peer-dep chain Bundle bloat if widgets include their own MUI copy
Medium Intentionally MUI-only internal ecosystems → Too restrictive for external widgets
MUI for host shell only
Shell chrome uses MUI; widgets opt in
Rich, polished shell UI out of the box No MUI dep required — widget authors choose their own stack Widget that adds @mui/material gets full useTheme() — ThemeProvider already in tree Widget without MUI still matches visually via CSS custom properties Single MUI copy in bundle (shell only, not duplicated) Accessible components, dark mode, responsive grid — free in shell
Widgets must respect the CSS token contract for visual harmony if not using MUI Widget authors who opt into MUI must pin a compatible version
Medium Projects with mixed or external widget authors ⭐ Recommended default
Design tokens only
CSS custom properties; no MUI at all
Maximum flexibility — any UI library in the shell Smallest bundle for a headless shell
Shell UI components must be built or sourced elsewhere More work to achieve accessible, polished chrome
Low Highly custom-branded shells, design systems already established → Premature unless you have an existing design system
Headless UI + tokens
Radix UI / Headless UI + custom styles
Full style control, zero opinionated CSS Accessible primitives (Radix UI) No design system lock-in
Significant styling work to reach MUI-level polish Visual consistency must be enforced by hand
Medium Teams with dedicated designers and existing token systems → Good Phase 2 complement for widget authors
💡 Two opt-in paths for widget authors — both are first-class:

Path A — plain React (no MUI): Widget reads CSS custom properties (var(--color-primary), var(--spacing-unit), etc.) that the shell mirrors from the MUI theme onto :root. No extra dependencies, zero bundle overhead, dark/light mode works automatically.

Path B — MUI in the widget (opt-in): Widget author adds @mui/material as their own dependency and calls useTheme() or uses any MUI component. Because the shell's <ThemeProvider> is already wrapping the widget in the React tree, the full live theme (colours, typography, spacing, breakpoints) is available with no extra wiring. MUI components render in the shell's palette automatically.

The shell does two things to enable both paths: (1) wraps the widget tree in <ThemeProvider theme={muiTheme}>; (2) runs a small effect that writes muiTheme values as CSS custom properties on :root.
⚠️ Avoid: Making @mui/material a peer dependency in the widget SDK package. The SDK defines the WidgetModule interface — it should have no UI library dependency at all. Widget authors who want MUI add it themselves; those who don't are not penalised.
Decision 5

Runtime vs Build-Time Inclusion

When does widget code join the running application? Build-time means the widget is part of the deployed bundle; runtime means it arrives in the browser after the shell has loaded.

Option Pros Cons Complexity Best Fit Recommendation
Build-time (static import)
All widgets bundled at deploy
Simplest, most reliable Full tree-shaking + optimization No runtime fetch failures Works offline / in air-gapped envs
Adding a widget requires a new deploy Bundle grows with every widget
Low Internal tools, controlled deployment ⭐ Recommended for MVP
Build-time lazy chunks
Code-split per widget, loaded on demand
Smaller initial bundle Still reproducible builds React.lazy / dynamic import — standard
Still requires rebuild to add widget Slightly more complex router/shell
Low Many widgets, perf-sensitive first load ✓ Use this from day one — easy win
Runtime remote ESM
import(url) from a CDN or widget server
No shell redeploy for new widgets Widget teams own their deploy cycle
React must be externalized and shared Import maps or global window trick needed Network dependency at runtime CSP policy must permit dynamic origins
Medium Independently-deployed widget teams → Phase 2
Module Federation
Webpack federated modules
Best shared-dep story for runtime Established in large monorepos
All teams must use Webpack (or compatible plugin) Build config overhead is significant
High Large teams already using MFE pattern → Avoid unless already committed
ℹ️ Practical path: Use React.lazy + dynamic import for code-splitting from day one. This gives you lazy loading for free within your bundle. When you're ready for runtime remote widgets, the import(url) call looks identical in code — you're just swapping a path for a URL.

Theming / Design tokens

Widget config API
Read/write own instance config		 Widget can persist its own settings without coupling to shell internals Low-Med — scoped writes MVP ⭐ Always include

Medium TypeScript-first teams ✓ Use Zod internally; convert to JSON Schema for storage
Custom config component
Widget ships its own config panel UI
Richest configuration UX possible Widget author has full control

💡 Design tip: Store dashboard state as a JSON document: { widgets: [{ id, type, layout, config }] }. The config field is widget-specific and opaque to the shell — validated only by the widget's own schema. This makes import/export trivial and keeps the shell decoupled from config internals.

Decision 9

Compatibility & Versioning

As the host dashboard evolves, how do you prevent breaking existing widgets? Getting this wrong means widget authors have to update their code every time the shell changes.

Option Pros Cons Complexity Best Fit Recommendation
No formal versioning
All widgets always match shell version
No overhead
Breaks the moment a widget is external No safe upgrade path
Trivial Internal monorepo, no external widgets ever → Fine only if truly internal forever
Versioned SDK package
widget-sdk@x.y.z as peer dep
SemVer communicates breaking changes Widget authors pin to a supported range Shell can support multiple SDK versions Types are always in sync with runtime
Requires discipline in publishing releases Major versions are still breaking for widget authors
Medium Any project with external widget contributors ⭐ Recommended — set up from day one
Adapter / shim layer
Shell translates old API calls to new
Old widgets keep working after shell upgrades Smooth migration path
Shims accumulate technical debt Must be tested against all supported widget SDK versions
Medium-High Mature ecosystems protecting existing widget authors → Phase 3 if SDK has breaking major version
Explicit API contract + changelog
Documented stable surface + deprecation notices
Sets clear expectations Low overhead, complements versioning Builds trust with widget authors
Not a substitute for SemVer — use both
Low All projects ⭐ Always do this alongside versioned SDK
⚠️ Common mistake: Teams defer creating the SDK package until they have external contributors. By then, the widget interface is already informally scattered across the codebase and painful to extract. Create the @yourproject/widget-sdk package as a local workspace package on day one — even if no one else uses it yet.

Roadmap

Phased Recommendation

Build the minimum viable foundation first, then add extensibility in phases. Each phase should be independently shippable and not require the next one to deliver value.

Phase 1 — MVP

Stable Foundation

  • SPA shell — Vite + React + React Router, component for widget grid
  • Widget SDK package — WidgetModule interface, WidgetMeta, typed config schema
  • Build-time widget loading — npm packages, code-split with React.lazy
  • Drag-and-drop layout — react-grid-layout or similar, serializable positions
  • localStorage persistence — JSON document: layout + per-widget config
  • Import / export dashboard — download / upload JSON file
  • Theme tokens — MUI theme provider in shell; CSS custom properties on :root for widget access; dark/light mode support
  • One external widget — separate repo, installed via npm to prove end-to-end
  • Trusted first-party model — no sandbox needed yet

Phase 2 — Next

Extensibility & UX Polish

  • Per-widget config panel — generic form rendered from widget's Zod schema
  • Multiple widget instances — same widget type, different configs
  • Runtime URL loading — import(url) with import maps for shared React
  • Capability API — typed HostContext passed to widgets instead of raw context
  • Widget template repo — GitHub template for external contributors
  • Event bus — typed pub/sub for cross-widget communication
  • Named dashboard tabs/pages — multiple dashboard views
  • SDK changelog + deprecation policy — semver with documented breaking changes

Phase 3 — Later

Scale & Ecosystem

  • Backend persistence — API + DB for cross-device sync
  • User accounts — auth, per-user dashboards, sharing
  • Shared data / query layer — widgets access app data via typed hook
  • Auth proxy for widgets — widgets make authenticated calls through shell
  • Web Component adapter — allow non-React widgets via custom elements
  • iframe sandbox option — opt-in for untrusted third-party widgets
  • SDK shim / adapter layer — support old widget SDK versions
  • Widget registry / catalog — if building an open ecosystem
Default Stack

Suggested Default Architecture

For a small-to-medium project that wants to start fast and stay extensible without overengineering, here is the concrete default stack. Every choice below has a clear upgrade path for when you outgrow it — none require a rewrite.

Build Tool
Vite ✅
Fast dev server, native ESM, easy code-splitting. Supports import maps for future runtime loading.
Language
TypeScript (strict mode) ✅
The SDK is a TypeScript contract. Strict mode catches interface drift early. Widgets written in TS get full autocomplete.
UI Framework
React 18+ ✅
Shared singleton peer dep in all widget packages. Suspense for lazy widget loading.
State Management
Jotai ✅
Atomic, bottom-up state — atoms compose well with the per-widget instance model. Minimal boilerplate, no context providers needed per widget. Shell can expose atoms via the HostContext for widgets that opt in.
Routing
None for MVP ✅
Single-view dashboard canvas — no routes needed. Add TanStack Router or React Router when a second view materialises (settings page, multi-tab dashboards, etc.).
TanStack RouterReact Router v6
Type safetyEnd-to-end — params, search, loaders all typedManual typing needed
Search params as stateFirst-class, typed, serializedVia useSearchParams — untyped by default
Community / docsSmaller but excellentMassive, tons of examples
Learning curveSteeper upfrontVery familiar to most devs
File-based routingOptional, built-inVia Remix conventions only
Verdict for this projectFor a TS-first project with few routes, TanStack Router pays off; React Router is the safe/familiar fallback.
Layout / DnD
react-grid-layout ✅
Battle-tested resizable drag-and-drop grid. Serialize layout as plain JSON.
react-grid-layoutdnd-kit
Grid snappingBuilt inYou build it
Resize handlesBuilt inYou build it
Layout serialization[{i,x,y,w,h}] — doneYou design the format
Collision detectionBuilt inYou build it

Widget SDK
@yourproject/widget-sdk (workspace pkg)
WidgetModule interface, WidgetMeta, config schema helper. Versioned with SemVer from commit zero.
Config Schema
Zod (in SDK)
Define config shape once. Get TypeScript types and runtime validation. Convert to JSON Schema for storage.
Persistence (MVP)
localStorage + JSON import/export
Single serializable dashboard document. Upgrade path: swap storage layer for API calls without changing the document schema.
Design System
MUI (shell) + CSS custom properties (widgets)
MUI powers the host shell chrome (app bar, drawers, dialogs). On theme load, MUI tokens are mirrored as CSS custom properties on :root so widgets can match the visual style without an MUI dependency.
Theming
CSS custom properties on :root
Widgets read tokens directly (e.g. --color-primary, --spacing-unit). Dark/light mode flips all tokens automatically. Works with any CSS-in-JS or plain CSS in widgets.
Widget Loading (MVP)
npm packages + React.lazy
Installed as dependencies. Code-split per widget. Upgrade path: replace with import(url) when ready.
Trust Model
Trusted first-party, capability-limited API
Widgets receive a typed HostContext object — they don't import from the shell directly. Makes future sandboxing feasible.
Distribution
Separate repo per widget family
Shell repo + one or more widget repos. Each publishes to npm (private or public). SDK is the shared contract.

The single most important structural decision: create the @yourproject/widget-sdk package before writing your first widget, and pass a typed HostContext to every widget rather than letting widgets import from the shell directly. This one constraint keeps the rest of the architecture honest and makes every upgrade path in this guide tractable.
